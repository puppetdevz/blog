    垃圾回收器与垃圾回收算法的关系：

    • 垃圾回收器是垃圾回收算法的具体实现
    • 垃圾回收器实现了多种垃圾回收算法
    • 垃圾回收器是系统垃圾回收的具体使用

    由于垃圾回收需要根据回收对象特征的不同、场景的不同，采用不同的垃圾回收算法，以达到效率最大化，因此垃圾回收器不仅仅实现了垃圾回收算法，还是采用某种策略将其组合起来。

    在 JVM 中有多种垃圾回收器，例如：串行垃圾回收器、并行垃圾回收器、CMS（并发）回收器、G1 垃圾回收器等等

    其中，G1 垃圾回收器是现在 JVM 默认采用的，需要重点了解

    # 串行垃圾回收器

    串行垃圾回收器，是指使用单线程进行垃圾回收，垃圾回收时，只有一个线程在工作， Java 应用中的所有其他线程都要暂停，等待垃圾回收的完成。

    这种现象称之为 STW（Stop-The-World）

    对于交互性较强的应用而言，这种垃圾回收器是不能够接受的。

    在 Java web 应用中是不会采用这种回收器的。

    # 测试代码

    public class DemoSerialGC {
    public static void main(String[] args) throws Exception {
        List<Object> list = new ArrayList<>();
        int times = 10000;
        while (true) {
            int sleep = new Random().nextInt(100);
            if (System.currentTimeMillis() % 2 == 0) {
                list.clear();
            } else {
                for (int i = 0; i < times; i++) {
                    Properties properties = new Properties();
                    properties.put("key_" + i, "value_" + System.currentTimeMillis() + i);
                    list.add(properties);
                }
            }
            // System.out.println("the size of list: " + list.size());
            Thread.sleep(sleep);
        }
    }
}

    # 并行垃圾回收器

    并行垃圾回收器在串行垃圾回收器的基础上做了改进：将单线程改为多线程，这样子可以缩短垃圾回收的时间（但机器需要支持并行，否则也不一定也会更快）

    并行垃圾回收器在运行过程中也会暂停应用程序，这点跟串行回收器是一样的，只是由于是并行执行，所以速度更快些

    # ParNew垃圾回收器

    ParNew 垃圾回收器工作在年轻代上，只是将串行的回收器改成了并行。

    通过 -XX:+UseParNewGC 参数设置年轻代使用 ParNew 回收器，年老代依旧为串行回收器

    # 为了测试GC，将堆的初始和最大内存都设置为16M 
-XX:+UseParNewGC -Xlog:gc* -Xms16m -Xmx16m
# 如果是低版本的 jdk，可能不支持-Xlog:gc*，用下面的命令代替
-XX:+UseParNewGC -XX:+PrintGCDetails -Xms16m -Xmx16m

    # ParallelGC垃圾回收器

    ParallelGC 回收器的工作机制与 ParNew 回收器相同，只是在 ParNew 回收器的基础上，新增了两个与系统吞吐量相关的参数，让它使用起来更加灵活和高效，相关参数如下：

    • -XX:+UseParallelGC：年轻代使用 ParallelGC 垃圾回收器，老年代使用 ParallelOldGC 垃圾回收器

    在 JDK 8 中有所不同：

    • -XX:+UseParallelGC：年轻代使用 ParallelGC 垃圾回收器，老年代使用串行回收器
    • -XX:+UseParallelOldGC：年轻代使用 ParallelGC 垃圾回收器，老年代使用 ParallelOldGC 垃圾回收器（新版本 JDK 已经移除该参数）
    • -XX:MaxGCPauseMillis
      • 设置最大的垃圾回收时的停顿时间，单位为毫秒
      • 需要关注的是，ParallelGC 为了达到设置的停顿时间，可能会调整堆大小或其他的参数，如果堆的大小设置的较小，就会导致 GC 工作变得很频繁，反而可能会影响到性能
      • 该参数的使用需要谨慎
    • -XX:GCTimeRatio
      • 设置 GC 时间栈程序运行时间的百分比，公式为：$1/(1+n)$
      • 值为 0 - 99 之间的数字，默认值为 99，即 GC 时间不能超过 1%
    • -XX:UseAdaptiveSizePolicy
      • 自适应 GC 模式，垃圾回收器将自动调整年轻代、老年代等参数，达到吞吐量、堆大小、停顿时间之间的平衡
      • 一般用于，手动调整参数比较困难的场景，让回收器自动进行调整

    # CMS垃圾回收器

    CMS 全称 Concurrent Mark Sweep，是一款并发的、使用标记 - 清除算法的垃圾回收器，该回收器是针对老年代垃圾回收的，通过参数 -XX:+UseConcMarkSweepGC 进行设置

    CMS 垃圾回收器的执行过程如下：

    • 初始标记(initial-mark), 标记 root，这时候会导致 STW

    • 并发标记(concurrent-mark)，此时用户线程同时运行，不需要挂起（这就解决了串行/并行垃圾回收器 STW 问题）

    • 预清理（concurrent-preclean），与用户线程同时运行

    • 最终标记(Final Remark)，会导致 STW，而之所以需要“最终标记”这个阶段，是因为在“并发标记”和“预清理”阶段用户线程仍在运行，这期间会继续产生内存垃圾

    • 并发清除(concurrent-sweep)，与用户线程同时运行

    • 调整堆大小（Resizing）：设置 CMS 在清理之后进行内存压缩，目的是解决内存碎片化问题

    • 重置（Resetting）：并发重置状态等待下次 CMS 的触发(concurrent-reset)，与用户线程同时运行

    # G1垃圾回收器（重点）

    G1 垃圾回收器是在 jdk1.7 中正式使用的全新的垃圾回收器，oracle 官方计划在 jdk9 中将 G1 变成默认的垃圾回收器，以替代 CMS（即在 Java 9 之前，默认的垃圾回收器是 CMS）

    有了 G1 收集器后，建议都使用 G1，前面学习的串行、并行和 CMS 回收器当做了解。

    在 G1 回收器之前，我们对 JVM 在垃圾回收方面的调优需要分别对年轻代、老年代进行调优，但是有了 G1 之后，我们的调优变得简单了许多。

    G1 的设计原则就是简化 JVM 性能调优，开发人员只需要简单的三步即可完成调优：

    • 第一步，开启 G1 垃圾回收器

    • 第二步，设置堆的最大内存

    • 第三步，设置最大的停顿时间

    G1 中提供了三种垃圾回收模式，Young GC、Mixed GC 和 Full GC，会在不同的条件下被触发

    # 原理

    G1 垃圾回收器相对比其他回收器而言，最大的区别在于它取消了年轻代、老年代的物理划分，取而代之的是将堆划分为若干个区域（Region），这些区域中包含了有逻辑上的年轻代、老年代区域。

    这样做的好处是，我们不需要单独地对每个代进行设置，不用担心每个代的内存是否足够

    重要的是下面这个图：

    在 G1 划分的区域中，年轻代（上面一整块都是年轻代，只是逻辑上划分为了 Eden、Survivor、Old 和 Humongous）的垃圾回收依然采用暂停所有应用线程的方式，将存活对象拷贝到 Old 或者 Survivor（Eden -> Survivor，Survivor -> Old），即 G1 的通过将对象从一个区域复制到另外一个区域，完成了清理工作（年轻代采用复制算法）

    这里需要重点注意！！！在上面的处理过程中，G1 就已经完成了部分堆的压缩，因为将存活对象复制到 Old 或 Survivor 后，原来的区域内存会被清理，变成一块连续可用的区域内存，这样可以解决内存碎片化问题。

    在 G1 中，有一种特殊的区域，叫 Humongous 区域。

    • 如果一个对象占用的空间超过了分区容量 50% 以上，G1 就认为这是一个巨型对象。

    • 这些巨型对象，默认直接会被分配在老年代，但是如果它是一个短期存在的巨型对象，就会对垃圾回收器造成负面影响。

    • 为了解决这个问题，G1 划分了一个 Humongous 区，它用来专门存放巨型对象。如果一个 H 区装不下一个巨型对象，那么 G1 会寻找连续的 H 分区来存储。为了能找到连续的 H 区，有时候不得不启动 Full GC。

    # Young GC

    Young GC 主要是对 Eden 区进行 GC，它在 Eden 空间耗尽时会被触发。

    • Eden 空间的数据移动到 Survivor 空间中，如果 Survivor 空间不够，Eden 空间的部分数据会直接晋升到年老代空间。
    • Survivor 区的数据移动到新的 Survivor 区中，也有部分数据晋升到老年代空间中。
    • 最终 Eden 空间的数据为空，GC 停止工作，应用线程继续执行。

    # Remembered Set（已记忆集合）

    在 GC 年轻代的对象时，我们如何找到年轻代中对象的根对象呢？

    根对象可能是在年轻代中，也可以在老年代中，那么老年代中的所有对象都是根么？

    如果全量扫描老年代，那么这样扫描下来会耗费大量的时间。

    于是，G1 引进了 RSet 的概念。它的全称是 Remembered Set，其作用是跟踪指向某个堆内的对象引用。

    每个 Region 初始化时，会初始化一个 RSet，该集合用来记录并跟踪其它 Region 指向该 Region 中对象的引用，每个 Region 默认按照 512Kb 划分成多个 Card，所以 RSet 需要记录的东西应该是 xx Region 的 xx Card

    # Mixed GC

    当越来越多的对象晋升到老年代 old region 时，为了避免堆内存被耗尽，虚拟机会触发一个混合的垃圾回收器，即 Mixed GC，该算法并不是一个 Old GC，除了回收整个 Young Region，还会回收一部分的 Old Region

    这里需要注意：是一部分老年代，而不是全部老年代，可以选择哪些 old region 进行回收，从而可以对垃圾回收的耗时时间进行控制。

    也要注意的是 Mixed GC 并不是 Full GC。

    MixedGC 什么时候触发？ 由参数 -XX:InitiatingHeapOccupancyPercent=n 决定。默认：45%，该参数的意思是：当老年代大小占整个堆大小百分比达到该阀值时触发。

    它的 GC 步骤分 2 步：

    1. 全局并发标记（global concurrent marking）

    2. 拷贝存活对象（evacuation）

    # 全局并发标记

    全局并发标记，执行过程分为五个步骤：

    • 初始标记（initial mark，STW）：标记从根节点直接可达的对象，这个阶段会执行一次年轻代 GC，会产生全局停顿。

    • 根区域扫描（root region scan）

      • G1 GC 在初始标记的存活区扫描对老年代的引用，并标记被引用的对象。
      • 该阶段与应用程序（非 STW）同时运行，并且只有完成该阶段后，才能开始下一次 STW 年轻代垃圾回收。

    • 并发标记（Concurrent Marking）：G1 GC 在整个堆中查找可访问的（存活的）对象。该阶段与应用程序同时运行，可以被 STW 年轻代垃圾回收中断。

    • 重新标记（Remark，STW）：该阶段是 STW 回收，因为程序在运行，针对上一次的标记进行修正。

    • 清除垃圾（Cleanup，STW）：清点和重置标记状态，该阶段会 STW，这个阶段并不会实际上去做垃圾的回收，等待 evacuation 阶段来回收

    # 拷贝存活对象

    Evacuation 阶段是全暂停的。该阶段把一部分 Region 里的活对象拷贝到另一部分 Region 中，从而实现垃圾的回收清理。

    # G1回收器相关参数

    • -XX:+UseG1GC：使用 G1 垃圾回收器

    • -XX:MaxGCPauseMillis：设置期望达到的最大 GC 停顿时间指标（JVM 会尽力实现，但不保证达到），默认值是 200 毫秒。

    • -XX:G1HeapRegionSize=n：设置的 G1 区域的大小。值是 2 的幂，范围是 1 MB 到 32 MB 之间。目标是根据最小的 Java 堆大小划分出约 2048 个区域。

      默认是堆内存的 1/2000。

    • -XX:ParallelGCThreads=n：设置 STW 工作线程数的值。将 n 的值设置为逻辑处理器的数量。n 的值与逻辑处理器的数量相同，最多为 8。

    • -XX:ConcGCThreads=n：设置并行标记的线程数。将 n 设置为并行垃圾回收线程数(ParallelGCThreads)的 1/4 左右。

    • -XX:InitiatingHeapOccupancyPercent=n：设置触发标记周期的 Java 堆占用率阈值。默认占用率是整个 Java 堆的 45%。

    # 回收器优化建议

    • 年轻代大小
      • 避免使用 -Xmn 选项或 -XX:NewRatio 等其他相关选项显式设置年轻代大小。
      • 固定年轻代的大小会覆盖暂停时间目标
    • 暂停时间目标不要太过严苛
      • G1 GC 的吞吐量目标是 90% 的应用程序时间和 10% 的垃圾回收时间。
      • 评估 G1 GC 的吞吐量时，暂停时间目标不要太严苛。目标太过严苛表示愿意承受更多的垃圾回收开销，而这会直接影响到吞吐量。
