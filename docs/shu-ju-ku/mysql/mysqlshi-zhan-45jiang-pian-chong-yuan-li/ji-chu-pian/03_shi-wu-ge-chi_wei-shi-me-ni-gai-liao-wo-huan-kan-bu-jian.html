    # 事务中四种隔离等级的原理

    事务中，有四种隔离等级

    • 读未提交
    • 读已提交
    • 可重复读
    • 串行

    这四种隔离等级中分别会有四种不同的数据读取情况

    • 在“读未提交”中，会出现“脏读”，即读取到其他事务中变更但未提交的数据，此时也是“不可重复读”的情况。
    • 在“读已提交”中，“脏读”不再出现，但能够读取到其他事务中变更且已提交的数据，此时也是“不可重复读”的情况
    • 在“可重复读”中，在事务开启到事务结束，读取到的数据都是一致的，即为“可重复读”，但此时会有“幻读”出现，例如事务 A 通过 select 确认不存在用户 puppet，然后想插入用户 puppet 的信息，但事务 B 在事务 A 插入用户 puppet 之前先插入了用户 puppet，此时事务 A 就会插入失败，但是在事务 A 中始终无法查询到用户 puppet 的信息，这就是幻读。
    • 在“串行”中，在事务开启到事务结束，读取到的数据都是一致的，即为“可重复读”，同时也不会有“幻读”。

    这四种不同的数据读取情况由以下原理实现：

    首先，数据库里面会创建一个视图，访问的时候以视图的逻辑结果为准

    • 在“读未提交”隔离级别下，直接返回记录上的最新值（磁盘上），没有视图概念；
    • 在“读已提交”隔离级别下，这个视图是在每个 SQL 语句开始执行的时候创建的。
    • 在“可重复读”隔离级别下，这个视图是在事务启动时创建的，整个事务存在期间都用这个视图。
    • 在“串行化”隔离级别下，直接用加锁的方式来避免并行访问。即对于同一行记录，“写”会加“写锁”，“读”会加“读锁”。当出现读写锁冲突的时候，后访问的事务必须等前一个事务执行完成，才能继续执行。

    # 事务隔离的实现

    以“可重复读”为例（MVCC：多版本并发控制，通过 undo log 版本链和 read-view 实现事务隔离）

    在 MySQL 中，实际上每条记录在更新的时候都会同时记录一条回滚操作。记录上的最新值，通过回滚操作，都可以得到前一个状态的值。（意思就是除了记录变更记录，还会记录一条变更相反的回滚操作记录，前者记录在 redo log，后者记录在 undo log）

    # 回滚日志的处理策略

    回滚日志总不能一直保留，什么时候删除呢？

    答案：在不需要的时候才删除。系统会判断，当没有事务再需要用到这些回滚日志时，回滚日志会被删除。

    # 为什么不建议使用长事务？

    长事务意味着系统里面会存在很老的事务视图。由于这些事务随时可能访问数据库里面的任何数据，所以这个事务提交之前，数据库里面它可能用到的回滚记录都必须保留，这就会导致大量占用存储空间，所以不建议使用长事务。

    在 MySQL 5.5 及以前的版本，回滚日志是跟数据字典一起放在 ibdata 文件里的，即使长事务最终提交，回滚段被清理，文件也不会变小。之前存在案例：数据只有 20GB，而回滚段有 200GB 的库。最终只好为了清理回滚段，重建整个库。

    除了对回滚段的影响，长事务还占用锁资源，也可能拖垮整个库。

    长事务带来的潜在问题：

    • 由于回滚日志一直被保留，可能会导致存储空间被大量占用
    • 由于长事务占用锁资源，可能会拖垮整个数据库

    # 事务的启动方式与长事务的误用

    长事务存在一些潜在风险，我们应该避免使用。但实际开发中，我们并非有意使用长事务，而是因为对 MySQL 的不够了解而误用了长事务。为什么这么说呢？事务的启动方式有两种：

    • 显式启动事务语句， begin 或 start transaction。配套的提交语句是 commit，回滚语句是 rollback。这也是我们最常用的方式，这种方式下，一般不会误用。

    • set autocommit=0，这个命令会将这个线程的自动提交关掉。意味着执行一个 select 语句，这个事务就启动了，而且并不会自动提交。这个事务持续存在直到主动执行 commit 或 rollback 语句，或者断开连接。

      这种方式就很容易导致误用长事务。甚至有些客户端连接框架会默认连接成功后先执行一个 set autocommit=0 的命令。这就导致接下来的查询都在事务中，如果是长连接，就导致了意外的长事务。（注意：长连接并不意味这长事务，还需要将 autocommit 设置为 0）

    因此，这也是为什么 begin 或 start transaction 的方式最为常用，这也是最为推荐的，不仅更具语义化，也可以避免误用长事务。

    # 查询现有长事务

    我们可以在 information_schema 库的 innodb_trx 这个表中查询长事务，比如下面这个语句，用于查找持续时间超过 60s 的事务。

    select * from information_schema.innodb_trx where TIME_TO_SEC(timediff(now(),trx_started)) > 60

    现在知道了系统里面应该避免长事务，如果我们是业务开发负责人同时也是数据库负责人，会有什么方案来避免出现或者处理这种情况呢？

    3. 通过配置参数 max_execution_time 指定事务执行的最长时间
